# Codex 使用 grok-build Skill 的编排规则

将本文件的规则合并到业务项目的 `AGENTS.md`，或在 Codex 会话开始时发送。`grok-build` Skill 负责通过预编译 wrapper 调用 Grok；Codex 仍是规划者和最终审计者。

## 工作流程

1. 理解需求并列出可验证的验收标准。
2. 调用前检查仓库状态，避免覆盖用户已有修改。
3. 把目标、约束、相关文件和验收标准整理成完整 prompt，调用 `$grok-build`。
4. 第一轮传入 `session_id: null`。保存结果中的 Grok UUID；返工时原样传回，wrapper 会使用 `--resume`。
5. `cwd` 使用项目根目录的实际路径；只有可信任务才设置 `auto_approve: true`。
6. Grok 执行期间不要并发修改相同文件。
7. 返回后独立执行 `git status --short`、`git diff --check`、`git diff`，并运行项目相关测试、lint、格式化和类型检查。
8. 审计失败时，明确写出证据、预期行为、涉及文件和验收标准，再用同一会话 UUID 返工。
9. 最多自动返工五轮；仍未通过则停止并报告根因与当前状态。
10. 不得删除或跳过测试、降低安全检查、吞掉异常，也不得自动 commit、push、merge、创建 PR 或执行不可逆操作。

## Prompt 模板

```text
你是本任务的具体实现者，请直接检查并修改当前项目。

目标：
<用户目标>

验收标准：
1. <可验证标准>
2. <可验证标准>

约束：
- 只修改与任务直接相关的文件。
- 保持现有公开接口兼容，除非目标明确要求改变。
- 增加或更新必要测试，并运行相关检查。
- 不要 commit、push 或创建 PR。
- 完成时报告修改文件、测试结果和未解决风险。
```

返工 prompt 应保留已经正确的实现，仅要求修复审计发现的问题及其直接原因。只有全部验收和检查通过后，Codex 才能向用户报告完成。
